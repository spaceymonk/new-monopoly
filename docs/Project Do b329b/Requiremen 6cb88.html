Introduction

This is the Requirements Specification document for a new web-based game which is called New Monopoly. This new game design will comprehensively upgrade the current Monopoly game and enable its users to play on a web browser. This document describes the system requirements with functional and non-functional requirements in addition to the use case model of the game.

Overview of Document

The overview of the document gives detailed specifications about our New Monopoly game. The organization of the document is done as below:

  1. System Requirements:
    • Functional Requirements: In this section, functional requirements are listed. Each functional requirement shows the required behavior by New Monopoly.
    • Non-Functional Requirements: In this section, non-functional requirements are organized and listed according to their categories. Each non-functional requirement shows a constraint or a requirement on the general characteristics of New Monopoly.
  1. Use Case Model: All user types and the scenarios for given user types are shown in lists. Also, all requirements of the system are shown with the help of use cases. Every use case shows how the relationship between the system and the user. Use cases are given with both explanations and with a diagram. The diagram shows all relations between use cases and user types.
    • User Types: All user types are shown as a list.
    • User Scenarios: Scenarios for each given user type are listed.
    • Use-Case Diagram: The use case diagram containing all use cases is shown.
    • Use Cases: All use cases are listed with their purpose, overview, actor(s), pre-condition(s), post-conditions(s), the alternative flow of events.

System Requirements

List of functional and non-functional requirements in natural language form.

Functional Requirements

  • The game shall include all functional requirements of a classic Monopoly board game.
  • The game bank shall have multiple currencies.
  • When a player decides not to buy a property, the property shall go on public auction by the system.
  • The game world shall run in 3D.
  • The game shall include a new game mode, where one player starts as the monopoly and the others as contenders.
  • The game economy shall be dynamic.
    • Property values shall be affected by inflation.
    • Currency exchange rates shall change based on the current economy.
    • The game should include future contracts.
  • The game shall allow the configuration of basic monopoly features.
    • The game shall allow players to enable a random sequence of spots as well as a periodic, random change of the properties pricing, rents, interests, and taxes rates (in both spots and cards).
    • The game shall allow a loyalty program to enable the use of virtual currency with random rules based on existing international alliances’ programs
    • The game shall allow new players to join the game as entrepreneurs (weaker restrictions on the number of players).
    • The game shall allow complexity setting in three levels (e.g., easy, medium, hard) based on rules and different configurations from exploring variabilities on the previous three bullets.

Non-functional Requirements

Usability

  • Users should be able to adapt to the system easily.
  • If a user knows how to play a classic Monopoly game, this system should present a familiar experience in the terms of pawns, cards, random events.

Reliability

  • The system must be operational for the duration of the whole game.
  • If there is an exception that results in the system shutting down, the recovery shouldn't take more than one minute.

Performance

  • The system should use a reasonable amount of internet.
  • The system's graphics need to be optimized to stabilize FPS.
  • The game logic of the system should be flawlessly executable.

Supportability

  • The system should be able to accommodate new updates without major reengineering.
  • The system's website shall be viewable from modern web browsers, such as Google Chrome, Apple Safari, and Mozilla Firefox.

Security

  • The system should enable its users to create and join rooms without requiring any sensible information.

User Documentation

  • There should be informative material that describes the newly added game mechanics.

Interface

The system shall be interfaced with:

  • Lobby creation scene
  • Lobby joining scene
  • Tutorial scene
  • The main game scene
  • End of the game scene

User Scenarios

Create personas and write narrative scenarios for the list of actions.

  • Player
    • As a player, I want to see my holdings even when it's not my turn so that I can work my strategy.
    • As a player, I want to see which holdings are already sold and to whom.
    • Monopoly
      • As a monopoly, I want to bankrupt all the other contenders and hold my monopoly.
    • Contenders
      • As a contender, I want to own a reasonable amount of the market.
      • As a contender, I want to bankrupt the monopoly or all the other contenders.
  • Lobby Creator
    • As a lobby creator, I want to share my lobby ID so that my friends.
    • As a lobby creator, I want to see who joined my lobby.
    • As a lobby creator, I want to start the game.
  • Lobby Joiners
    • As a lobby joiner, I want to join a game I know the ID of.
    • As a lobby joiner, I want to see who else is in the lobby.

Use Cases

Specify a use case for each feature/requirement in your project plan also containing alternative flows.

  1. Use Case Name: Join Lobby
    • Purpose: To play games with others, the user must join a lobby
    • Overview:
      1. A player starts the application.
      1. The system asks for the lobby name to join.
      1. The user enters the lobby name to join.
      1. The system checks if the lobby exists and if it does if it is full.
      1. The system includes the person in the game.
      1. The system asks for a user name.
      1. The player enters the user name.
      1. The system checks if the user name is taken by any other player in the lobby.
    • Actor(s):
      • Lobby Joiner
    • Pre Condition(s):
      • The player that is joining the lobby must have an internet connection.
      • The player must know the lobby name to join.
      • The lobby must exist so the player can enter.
    • Post Condition(s):
      • The player is included in the game.
    • The Alternative Flow of Events:
      • Step 4: If the lobby name doesn't exist or when it does if it is full, the use case goes back to step 2.
      • Step 8: If the user name is already taken, the use case goes back to 6.
  1. Use Case Name: Create Lobby
    • Purpose: For a game to be played, a user must create a lobby to play the game in.
    • Overview:
      1. A player starts the application.
      1. The system asks for a lobby and user name.
      1. The user enters the username.
      1. The system verifies the lobby name against all other lobbies.
      1. The system starts a game session and enters the lobby creator into the game.
    • Actor(s):
      • Lobby Creator
    • Pre Condition(s):
      • The lobby creator must have an internet connection.
    • Post Condition(s):
      • A game session has begun and other players can join to play the game.
    • The Alternative Flow of Events:
      • Step 4: If the lobby name is already taken, the use case goes back to step 2.
  1. Use Case Name: Tutorial
    • Purpose: Newcomers should be able to understand the concept that presented in the game.
    • Overview:
      1. The user starts the system.
      1. The user joins a lobby.
      1. The system will ask if the tutorial should be shown.
      1. If the user accepts, a tutorial will begin.
    • Actor(s):
      • Players
    • Pre Condition(s):
      • A lobby must be created.
      • The player should know the lobby name to join the lobby.
      • The user must have entered their user name and fully joined the game.
    • Post Condition(s):
      • The user should understand the basics of the game.
    • The Alternative Flow of Events:
      • Step 3: The user may not want to see the tutorial. If so, the business logic continues.
  1. Use Case Name: Start a New Game
    • Purpose: To enjoy the experience, the system should start the game.
    • Overview:
      1. The lobby creator presses start.
    • Actor(s):
      • Lobby Creator
    • Pre Condition(s):
      • Players must have an active internet connection.
      • The system should bear the load of the requests that arise from the game.
    • Post Condition(s):
      • The game must successfully launch.
    • The Alternative Flow of Events:
      • Step 1: Players might not connect to the game, the system will wait until everyone is connected. If any one of the players doesn't connect, the system will abort the game.
  1. Use Case Name: New Monopoly Gameplay
    • Purpose: After the lobby is created and the game has commenced, users will play a round of New Monopoly.
    • Overview:
      1. All players connect to the game after loading the screen.
      1. The system sets up players' turns randomly and the first player's turn begins.
      1. In each turn, two dice are rolled and the player whose turn is up moves according to the rolled dice.
      1. The game finishes if either the contenders take down the Monopoly or the Monopoly takes down the contenders.
    • Actor(s):
      • Players
    • Pre Condition(s):
      • A lobby must be created.
      • Players must be connected to the game.
    • Post Condition(s):
      • The game has ended in a definite result (Contenders or Monopoly win).
    • The Alternative Flow of Events:
      • Step 3: If a player disconnects from the game, they will be counted as AFK and their turns will be passed immediately after the dice roll.
  1. Use Case Name: Finishing a Game
    • Purpose: To find the winner and the loser the game must end.
    • Overview:
      1. When contenders or the monopoly wins, the game will end and the end-game screen will open.
      1. In the end-game screen, players will be able to see their performances and which side has won.
      1. The players will be able to "Play Again" and join a new lobby.
    • Actor(s):
      • Players
    • Pre Condition(s):
      • A game must be finished with one side (The monopoly or the contenders).
    • Post Condition(s):
      • A list of winners and losers is presented to the players.
    • The Alternative Flow of Events:
      • Step 3: If a player disconnects after the game, a new lobby will not be created.
