1. Introduction

This is the Design Specification document for a new web-based game which is called New Monopoly. This new game design will comprehensively upgrade the current Monopoly game and enable its users to play on a web browser. This document describes the system architecture with system high-level architecture and components diagrams while also explaining the low-level design of the game using class, sequence, and data flow diagrams.

2. System Architecture

2.1 System High-Level Architecture Diagram and Explanation

  • The above diagram shows the System High-Level Architecture Diagram for our New Monopoly Game. The diagram is explained step by step below.
  • Clients are players, whose numbers can change between 1 to 6.
  • Clients send their current status i.e. player info to the database.
  • Information of all players is sent synchronously to the database.
  • The database contains session data.
  • Database updates the game status of a game session.
  • Database updates the inflation rate and exchange rates between currencies of the bank.
  • The server contains the information of the game session and the bank.
  • The server sends the current status i.e. game info to all clients.
  • This whole process is kept throughout the duration of the game.

2.2 Component (Package) Diagram

  • The above diagram shows Component Diagram for our New Monopoly Game. Each relationship between components is explained in detail below.
  • Session Data is the main database that holds the current session’s data.
    • Session Data provides the players’ information to Game Session Component
  • Game Session provides current turn’s data to the Player component.
  • The player component uses Session Data, Property and Bank components
    • The player uses the Bank component for money transactions.
    • The player uses the Session Data component to get information about the current status of the game.
    • The player uses Property component to buy/sell properties.
  • Bank provides Inflation Rate to Property component. Because of this, its price can be dynamic.
    • Bank also uses Game Session to get information about the status of the game to change exchange rates and inflation rates according to it.

3. Low-Level Design

3.1 Class Diagrams

  • The above diagram shows all the class diagrams for our New Monopoly Game. Each class diagram is represented from top to bottom with its class name, class fields, and class methods.
  • Property Class
    • Fields:
      • propertyId: Id of the property that shows which zone of the game the property belongs to.
      • propertyPrice: Price of the property.
      • propertyColor: Color of the Property set (Such as Blue-Tarabya).
      • houseHotelCount: Shows how many house(s)/hotel(s) the property has.
      • rentAmount: Shows how much a player has to pay when stepping in that zone.
    • Methods:
      • propertyBough(): Method used to check whether a property is bought.
      • houseBuilt(): Method used when a house is built by a player.
      • hotelBuilt(): Method used when a hotel is built by a player.
      • propertyMortgaged(): Method used when a player mortgages their property for money.
      • checkColorSet(): Method used to check if the full set of zones is bought by a player.
  • Player Class:
    • Fields:
      • playerId: Id of the player.
      • playerName: Name of the player.
      • dollarAmount: Total dollar amount of the player.
      • euroAmount: Total euro amount of the player.
      • liraAmount: Total lira amount of the player.
      • properties: Properties the player has bought.
      • cards: Cards the player is holding.
    • Methods:
      • rollDice(): Method used to roll the dices.
      • move(): Method used to move on the board.
      • useJailFree(): Method used to use the jail-free card.
      • drawCard(): Method used to draw cards when stepped on specific zones.
      • payRent(): Method used to pay rent when stepping on zones of other players.
      • buyProperty(): Method used to buy a property.
      • buildHouse(): Method used to buy a house.
      • buildHotel(): Method used to buy a hotel.
      • joinLobby(): Method used to join a lobby.
      • createLobby(): Method used to create a lobby.
  • Game Session:
    • Fields:
      • thisTurn: Field to determine whose turn it is.
      • timeLeft: Field to determine how much time a player has left to play.
      • winnerParty: Field to show who has won.
      • thisSession: Field to show which session it is.
    • Methods:
      • startSession(): Method to start a session.
      • endSession(): Method to end a session.
      • nextTurn(): Method to go to the next turn.
      • roundTimer(): Method to provide timer for each turn.
      • checkWinCondition(): Method to check if any party has won.
  • Bank Class
    • Fields:
      • dollarAmount: The amount of dollar currency the player holds.
      • euroAmount: The amount of euro currency the player holds.
      • liraAmount: The amount of lira currency the player holds.
      • inflationPercentage: The inflation percentage is determined by the total amount of currencies each player has. Is used to raise property prices.
    • Methods:
      • giveMoney(): Method used to transfer money from the bank to a player
      • takeMoney(): Method used to transfer money from a player to the bank.
      • calculateInflation(): Method used to calculate the inflation rate based on the total number of currencies in the market.
      • calculateCurrencies(): Method used to determine exchange rates between each combination of currencies.

3.2 Sequence Diagrams

3.2.1 Diagram Containing All Use Cases

  • The above diagram shows the sequence diagram containing all 6 use cases for our New Monopoly game. The use cases include “Join Lobby”, “Create Lobby”, “Tutorial”, “Start a New Game”, “New Monopoly Gameplay” and “Finishing a game”. All these use cases are explained one by one below. One thing that can’t be seen in the below diagrams is the Lobby Loop which encompasses 5 use cases.

3.2.1.1 Use Case 1: Join Lobby

  • To join a lobby, Lobby Creator and Lobby Joiner invoke JoinLobby().
  • “Lobby Join Response” condition block represents two possible outcomes: Successful Response and Unsuccessful Response.
  • Depending on if the outcome is successful/unsuccessful, players that invoked JoinLobby() join/don’t join the lobby.

3.2.1.2 Use Case 2: Create Lobby

  • To create a lobby, Lobby Creator invokes CreateLobby().
  • “Lobby Creation Response” condition block represents two possible outcomes: Successful Response and Unsuccessful Response.
  • Depending on if the outcome is successful/unsuccessful, players that invoked CreateLobby() create/don’t create the lobby.

3.2.1.3 Use Case 3: Tutorial

  • To see the tutorial, the Player invokes PLayTutorial().
  • The player then sees the tutorial of the game.

3.2.1.4 Use Case 4: Start a New Game

  • To start the game, Lobby Creator invokes StartGame().
  • “Start Game Response” condition block represents two possible outcomes: Successful Response to Player actor and Unsuccessful Responses. The responses are represented to the Lobby Joiner and the Lobby Creator actors.
  • After StartGame() is invoked, the game is started.

3.2.1.5 Use Case 5: New Monopoly Gameplay

  • “Game Loop” condition block represents the gameplay of the game.
  • Player plays the game and the Game object checks if the game has ended or not.
  • Depending on if the game has ended or not, the player will keep playing them or the game will finish.

3.2.1.6 Use Case 6: Finishing a Game

  • To finish the game and show the scores, the Game object invokes FinishGame() and DisplayScore().
  • After FinishGame() and DisplayScore() are invoked, the game is finished and the scores are displayed.

3.3 Data Flow Diagram

  • The above diagram shows the Data Flow Diagram for our New Monopoly Game. The diagram is explained step by step below. The “Game Session Control” part is specifically made to show the level 2 DFD diagram.
  • Bank entity controls money transactions.
    • Give Money process is responsible for money flow from the Bank entity to the Player entity.
    • Take Money process is responsible for money flow from the Player entity to the Bank entity.
    • The Bank entity sends this information to the Game Session.
  • The Game Session Control process has 3 subtasks.
    • It controls the lobby operations.
    • It sets up the game settings
    • It also checks the money flow between players and the bank.
  • The player entity has can join or create a lobby.
    • This information of players is used to set up the game settings of the session.
    • The game is started after the game settings are set up.
  • The player entity can also buy properties.
    • Property entity updates Player’s properties and also
    • Property entity also sets own Property settings such as owner_id.
