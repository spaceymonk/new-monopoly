1. Introduction

  • This is the Test Specification document for a new game which is called New Monopoly. In this document, the goal is to step by step explain the testing strategies and results we had throughout the creation of the game. This document describes our general testing plan, our outlook on different additional testing requirements, and our final test results. In the testing plan, our general testing strategy and its subjects are explained while also explaining white/black box testing’s contribution to our project. We also explain additional testing strategies required for our project; such as security, performance, load/stress, and acceptance testing. Finally, we finish the document by providing a detailed explanation of our testing results.

2. Test Plan

2.1 Testing Strategy

  • Our main testing strategy was an incremental bottom-up approach. We used this testing strategy to test our project in small parts to then combine these parts to create a bigger architecture that we would then test again. The main three methods used were:
    • Unit Testing
    • Integration Testing
    • System Testing
  • Our way of completing our project was that we would assign each group member a task, and then when all of us finished those tasks; we would then combine them to create a bigger code architecture. Because of our system of implementation, we chose a bottom-up incremental approach.
  • The New Monopoly had many functions that we had to create and there were many small things that needed to work together after their initial implementation. Because of this, we thought an incremental approach would be best since any effect on the smaller subtasks would affect the entirety of the bigger architectures. We thought that by using an incremental approach, we would be able to test our tasks easier and also find flaws that could affect our system early. The reason we used a bottom-up approach was also quite the same. Since we would create small models as tasks first, we thought it would be better to test these models first and then integrate them. We also structured our testing system according to our way of doing the project.
  • For each task we assigned to a group member, we would do unit testing on it since it was the smallest piece of code architecture. Then when all these tasks were finished, we would combine them to create a bigger architecture of code that we would do integration testing on. We did this quite often as we dissected the project into many small code parts.
  • Since there were many tasks given to each individual member, there were many integration tests done on bigger code architecture parts. When all tasks were finished and they were combined with their respectively given tasks, those bigger code architectures had to be combined as well. This is where we used system testing.
  • We combined the bigger code architectures that were created by the combination of the smallest code parts. This was done at the end of the projects to finalize the project. On the biggest and final architecture of our project, we did system testing to test if everything we had done previously was able to work together.
  • We also did regression tests on code architectures that required integration and system testing.
  • Since our project was creating game and included a lot of both backend and frontend, we also used white-box and block-box testing.

2.2 Test Subjects

  • In the above figure, the Components Diagram for our New Monopoly Game can be seen. In this section, we will explain the data subject to integration testing from the five main components in the Components Diagram which are Property, Bank, Player, Game Session, Session Data. The list of data subject to integration testing can be seen below for the applicable components.
  • Property Component:
    • Rent Payment: The rent payment can differ throughout the game. Because of this, it is tested with many different features. The ownership of one property, the whole card set, properties with houses, properties with hotels, the money transaction system, etc. are some of the many different testing the rent payment system is subjected to.
    • Buying and Selling Properties: Properties are bought and sold on different occasions in the game. Buying/Selling a property is tested with transactions for each action, property availability, different decisions on available properties (buy or pass), and property selling on occasions where money is needed for a player.
    • Integration with Other Components: Properties are closely related to players and the bank. Property operations are done by players while all transactions are done with the help of the bank. Because of this, the property component is closely tested together with the bank and player components.
  • Player Component:
    • Turn Actions: Each turn consists of different three different game states which are dice roll, player movement, and the active state. The player is directly involved in two of these states which are dice roll and action state, player movement is done automatically. The states that the player is involved are tested. Dice roll mechanics, user buttons for a dice roll, property buying decisions, currency exchange point decisions, etc. are tested.
    • Money Transaction: Each player has money in three different currencies and can use this money on many different occasions such as buying properties, paying rent, exchanging money, etc. These types of operations are tested to create the general money flow of the game.
    • Integration with Other Components: Players are closely related to all other components since they are the ones creating all the actions in the game. Also, their data has to be stored throughout the game for the continuation of the game. Because of this, the player component is closely tested together with all other components.
  • Bank Component:
    • Inflation rate: The New Monopoly game has an inflation mechanic which, as the game progresses, creates inflation on the property prices. The mechanic is tested for the correct calculation of the interest rate and pricing calculations.
    • Currency rates: The current spots in the game allow players to exchange their money between different currencies. The bank is tested for the calculation of the currency rates and the conversion of players’ money.
    • Integration with Other Components: The bank is closely related to players and the game session. All the general money flow is done with the help of the bank and it's the banks’ job to deliver the changing status of the money in the game to the game session. Because of this, the bank component is closely tested together with the player and game session components.
  • Game Session Component:
    • There are interchangeable things for any game session. These are turn status, remaining time, dice roll, and game over the condition.
      • Turn Status: In each turn, players have a game screen for their actions and the player data. The actions of the turn are tested along with the data of the player which includes players’ time remaining, properties, and all three currency amounts.
      • Remaining Time: Each turn has a counter of sixty seconds per turn. The change of players’ turns according to the remaining time has been tested.
      • Dice Roll: In each turn, a player has to roll a dice to move around the map. The randomness of the dice roll mechanic along with the verification of dice data with movement has been tested.
      • Game Over Condition: The game ends when either the monopoly or all the remaining players go bankrupt. This condition has been tested for both outcomes.
      • Integration with Other Components: The game session is closely related to the bank and the player. While storing some data on its own, the game sessions get data from the bank and along with its data; supply it to the player. Because of this, the game session is closely tested together with the bank and player components.

Testing Tool(s)

For our tests, we don’t use extra testing tools since we use Unity. It is really easy to compile our source code and be able to see both the user interface and the terminal log for our test. Since Unity provides all that is needed, we use Unity standard compilation, UI and terminal for testing purposes.

3. Additional Tests

3.1 Security Testing

  • Security testing is a software testing mechanism that checks whether the system keeps sensitive data in a secure manner that prevents the information from getting into any intruder's hands or not. This testing is meant to reveal the weaknesses of the system and get in the way of any possible compromisations and data loss.
  • “Anti-Cheat Toolkit 2021” from Unity Asset Store can be used to test the general security of the game. This tool is used to discourse cheating, protect data, detect time cheating performances, etc. It is an all-around tool to check for any kind of cheat in a game. Test cases that can be used are listed below.
    • Protection of Code integrity: The tool is able to check code with the hashes generated, which makes sure the compiled code is not altered. This is used to test the integrity of the code.
    • Protection of Variables in Memory: Each player has many different variables and data that are kept in a game session such as money, properties, houses, etc. Protecting variables in the game session is important so that users can not change these important values to alter with the game. The tool can be used to check whether these types of altercations are done in the game.
    • Detection of Time Cheats: The game is run on a turn-based system that includes a timer for each turn. This tool can be used to test the turn timer to detect time cheats that might occur in-game where a user alters the time given for each turn.
    • Detection of Internet Attacks: It is important that the players get to play the game normally and not have to deal with attempts that would alter their state of connection to the game. The tool is also able to detect general cheating attempts such as kicking players from the game, DDoS attacks, etc. Such tests can be done to kick/ban players from the game.
    • General Security Measures for Our Game: The tool kit's source code is provided to users so its functionalities can be expanded to further include monopoly-specific anti-cheat functionalities.

3.2 Performance Testing

  • Performance testing is the most important type of testing for a game. If a game is not optimized and works on low frame rates, playing the game is almost impossible.
  • Unity has built-in tools for testing in-game performance for developed games. These tools are used for optimizing FPS values, memory allocation, CPU/GPU usage, etc.
  • We have used Unity Profiler: A tool that shows which part of the game uses which resource and how much. It’s simply an all-in-one testing tool to see what goes wrong. Test cases that we used with Unity Profiler are listed below.
    • Script Optimization: Using Profiler, we have analyzed our FPS (Frames Per Second) values. The values were mostly ideal (500+ FPS). But some scripts in our code spiked the CPU Usage and caused FPS drops. We have analyzed and tested these scripts using Profiler.
    • Physics Engine Performance: Dice roll system used 3D Physics and Physics use a lot of CPU if not optimized well. Unity Profiler also gives us the data of Physics Engine usage, so we also tested our Dice physics performance.
    • Memory Usage: Memory allocation is another piece of information that Profiler provides. We have tested if there is a memory leak or not using the Profiler.
    • GPU and CPU Usage: For a 3D game, rendering is the main issue for performance issues. GPU and CPU usage should be kept low with optimal scripting methods. We have tested our scripts and their optimization using Profiler’s CPU and GPU usage data.
  • Another tool used in performance testing was Frame Debugger, a built-in tool in Unity. Frame Debugger is used for optimizing the visual data of the game. It shows the rendering order on each frame, and how much each renders costs individually. Test cases that we used with Frame Debugger are listed below.
    • GPU Usage for Game Visuals: We have tested FPS values and GPU usage for each rendered visual in our game. These included but were not limited to:
      • Dice visuals
      • 3D design for the player pawns
      • Game board
      • Gamecards
      • Game screens
    • GPU usage for Game Background: We also added a new customized Skybox to our game for better background visuals, which would affect the game’s performance. We tested it with a frame debugger if it has a huge effect on our GPU usage.

3.3 Load/Stress Testing

  • Load testing is a type of performance testing mechanism which inspects the system behavior under increasing loads, meaning the number of users, data, operations hours, etc. are reaching maximum values. concurrently active.
  • “Photon Engine” is a “real-time socket server” and a multiplayer game development framework that can be used to test many different performance aspects such as connection analysis, server performance, etc. A photon server can be used to test these aspects. Test cases that can be used are listed below.
    • Session Handling: Each game session should be able to handle four players. The load of the current users on the system can be tested on photon servers.
    • Total Room Count: In a multiplayer web-based scenario, there will be many rooms with each having four players. The performance load of many rooms and users can be tested on photon servers.
    • Response Time: With many rooms and players, the load on the system increases. Even with this load, the system should be able to respond to each session in a proper time for the games to continue in normal order. Response time for the game sessions can be tested on the servers.
    • Session Data: As the game progresses, the data for each player will continue to accumulate. Players will have more money, properties, houses, etc. The server should be able to handle the game session data throughout the entirety of the game.
    • Session Time: Monopoly is a fairly long-lasting game. A game of monopoly can average 30 to 90 minutes. With this in mind, the servers should be able to be up and running throughout the entire game without any failures of connection. This can be tested using the photon servers.
    • User Interaction: Throughout the game, players will have to interact with the game using the UI to buy properties, exchange money, roll dice, etc. The server should be able to accept the requests by the users in session in etiquette time.

  • We observed some spikes on our Roll Dice Scripts and the system. As you can see on the profiler image above, physics (orange) has caused a big spike while the dice was rolling down on board. GPU and CPU Usage:

  • We have tested our game’s overall performance by monitoring the CPU and GPU usage at all times. GPU usage, combined with frame debugger gave us definite solutions for problems caused by visual rendering of the objects.

4.3.2. Frame Debugger:

  • Frame Debugger was another tool that we have used in Unity to check the render order and performance affections of every single visual batch the game has. Unity has a concept called “Draw Call”, every single visual in game is “drawn” on the screen. Some objects cause higher “Draw Calls” because they are more complex.
  • We optimized our UI and HUD elements to reduce our draw call for higher performance in our game. As you can see below, “UGUI.Rendering.RenderOverlays” screen has 26 “Draw Calls”, with optimization, we have reduced this number to 8. This didn’t affect our performance that much since 26 is still a relatively low number of “Draw Calls” for an actual game, but we tried to get the best performance as we could.
