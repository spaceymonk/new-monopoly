Introduction

This is the Requirements Analysis document for a new web-based game which is called New Monopoly. This new game design will comprehensively upgrade the current Monopoly game and enable its users to play on a web browser. This document describes the system requirements with functional and non-functional requirements, use case model, user interface model, and the flow diagrams of the game.

System Requirements

Functional Requirements

IdDescription
1The game shall include all functional requirements of a classic Monopoly board game.
2The game bank shall have multiple currencies.
3When a player decides not to buy a property, the property shall go on public auction by the system.
4The game world shall run in 3D.
5The game shall include a new game mode, where one player starts as the monopoly and the others as contenders.
6The game economy shall be dynamic.
7Property values shall be affected by inflation.
8Currency exchange rates shall change based on the current economy.
9The game should include future contracts.
10The game shall allow the configuration of basic monopoly features.
11The game shall allow players to enable a random sequence of spots as well as a periodic, random change of the properties pricing, rents, interests, and taxes rates (in both spots and cards).
12The game shall allow a loyalty program to enable the use of virtual currency with random rules based on existing international alliancesâ€™ programs
13The game shall allow new players to join the game as entrepreneurs (weaker restrictions on the number of players).
14The game shall allow complexity setting in three levels (e.g., easy, medium, hard) based on rules and different configurations from exploring variabilities on the previous three bullets.

Use Cases

Use-Case NameJoin Lobby
PurposeTo play games with others, the user must join a lobby
Overview1. A player starts the application. 2. The system asks for the lobby name to join. 3. The user enters the lobby name to join. 4. The system checks if the lobby exists and if it does if it is full. 5. The system includes the person in the game. 6. The system asks for a user name. 7. The player enters the user name. 8. Post Condition(s)1. The player is included in the game.

PurposeFor a game to be played, a user must create a lobby to play the game in.
Overview1. A player starts the application. 2. The system asks for a lobby and user name. 3. The user enters the username. 4. The system verifies the lobby name against all other lobbies. 5. The system starts a game session and enters the lobby creator into the game.

Pre Condition(s)1. The lobby creator must have an internet connection.
Post Condition(s)1. A game session has begun and other players can join to play the game.
The Alternative Flow of EventsStep 4: If the lobby name is already taken, the use case goes back to step 2.

Overview1. The user starts the system. 2. The user joins a lobby. 3. The system will ask if the tutorial should be shown. 4. If the user accepts, a tutorial will begin.
Actor(s)Players
Pre Condition(s)1. A lobby must be created. 2. The player should know the lobby name to join the lobby. 3. The user must have entered their user name and fully joined the game.
Post Condition(s)1. The user should understand the basics of the game.
The Alternative Flow of EventsStep 3: The user may not want to see the tutorial. If so, the business logic continues.
Use-Case NameStart a New Game
PurposeTo enjoy the experience, the system should start the game.
Overview1. To enjoy the experience, the system should start the game.
Actor(s)Lobby Creator
Pre Condition(s)1. Players must have an active internet connection. 2. The system should bear the load of the requests that arise from the game.
Post Condition(s)1. The game must successfully launch.
The Alternative Flow of EventsStep 1: Players might not connect to the game, the system will wait until everyone is connected. If any one of the players doesn't connect, the system will abort the game.
Use-Case NameNew Monopoly Gameplay
PurposeAfter the lobby is created and the game has commenced, users will play a round of New Monopoly.
Overview1. All players connect to the game after loading the screen. 2. The system sets up players' turns randomly. 3. The first player's turn begins. 4. Two dice are rolled in each turn. 5. The player whose turn is up moves according to the rolled dice. 6. The game finishes if either the contenders take down the Monopoly or the Monopoly takes down the contenders.
Actor(s)Players
Pre Condition(s)1. A lobby must be created. 2. Players must be connected to the game.
Post Condition(s)1. The game has ended in a definite result (Contenders or Monopoly win).
The Alternative Flow of EventsStep 5: If a player disconnects from the game, they will be counted as AFK and their turns will be passed immediately after the dice roll.
Use-Case NameFinishing a Game
PurposeTo find the winner and the loser the game must end.
Overview1. When contenders or the monopoly wins, the game will end and the end-game screen will open. 2. In the end-game screen, players will be able to see their performances and which side has won. 3. The players will be able to "Play Again" and join a new lobby.
Actor(s)Players
Pre Condition(s)1. A game must be finished with one side (The monopoly or the contenders).
Post Condition(s)1. A list of winners and losers is presented to the players.
The Alternative Flow of EventsStep 3: If a player disconnects after the game, a new lobby will not be created.

5.2 General data model

Above is the Entity-relationship diagram for the New Monopoly game. The game consists of 6 players per game session where each player has their own money/properties etc. Each game has a difficulty; easy, medium, or hard. The "Game Session" table is reflected to show these attributes in the game. There is also, if game mode is chosen this way, a monopoly in the game where all other players are trying to take down. Each game session has a number of players and a bank. Each player has their own money. This money is in dollars, euros, and liras. Each player can be a monopoly or a regular player and can also have a jail-free card or not. The "Player" table shows all these attributes. Each player also has the ability to buy a number of properties. Each property has its unique name and the players have the ability to be able to build houses and hotels on these properties as reflected by the "Properties" table. There is also a bank in the game. The bank is designed as the information center of the game as it controls the exchange and inflation rates that will be applied throughout the game. These exchange rates are rated between the dollar, euro, and lira. The "Bank" table is designed to represent all these attributes.

5.3 Important data considerations

Almost all of the code in our project will be developed in Unity except for the server-side. The most important problem here is the connection between the players on the server-side. To tackle this issue, we will be using "Colyseus", which is a Node.js-based multiplayer framework that enables us to synchronize server game states to all the players. Other than this, all codes will be written in Unity using C#.

5.4 Data flow

New Monopoly Game has a basic database structure. There are two main entities, "Player and "Game session". Each game session is created by a player using the process "Create lobby". Also other players can join a game session with the process "Join Lobby", up to a maximum of 6 players. The Player has a process of "Buy Property" which updates consisting property's owner_id to the buyer's user_id. Whenever a player buys a property, their money of chosen currency will be reduced by the amount of property's price. Bank has the processes "Give Money" and "Make Money", which update the liquidity data of players positively or negatively, respectively. Players are accessed through the Game Session for these processes to be carried out.
